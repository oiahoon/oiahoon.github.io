# 文章页 Product + UI/UX 专项评审

**更新日期**：2026-04-19  
**评审页面**：`https://notes.miaowu.org/articles/`  
**评审角色**：Product Manager、UI/UX Designer、Sr. Front-end Engineer、QA Reviewer、Code Reviewer  
**评审目标**：让文章页保持克制、安静、偏纸面归档的阅读气质，同时补足首屏语境、内容发现和移动端阅读效率。

## 1. 团队分工

### Product Manager

负责判断文章页是否清楚回答三个问题：

- 这里是什么内容？
- 新访客应该从哪里开始读？
- 内容增长后还能不能被有效发现？

当前判断：文章页作为个人博客的主阅读入口已经成立，但仍偏“漂亮的时间归档”，对新访客的内容引导不足。

### UI/UX Designer

负责判断页面是否在视觉上服务阅读，而不是制造装饰噪音。

当前判断：页面气质清爽、留白充足、文字层级稳定；但首屏留白略多，移动端标题视觉重量偏大，普通内容页 hero 区域的图标可继续做减法。

### Sr. Front-end Engineer

负责把方案落到 Astro 页面、collection 数据统计、响应式样式和主题兼容，不改变 Markdown + Astro 静态站架构。

实施原则：优先改低风险的文案、间距、图标和标签入口；不引入 CMS、不做 infinite scroll、不把文章列表改成重卡片流。

### QA Reviewer

负责验证桌面、平板、手机、主题切换、内容样本和上线后真实页面。

验收重点：浅色/深色可读、移动端不横向滚动、长标题不破坏布局、标签入口可访问、构建和内容检查通过。

### Code Reviewer

负责检查改动范围、可维护性和回归风险。

审查重点：不把文章页专项改动泄漏到摄影页；共享样式如需修改，必须列出跨页面回归范围；不新增重型依赖或动态渲染成本。

## 2. 当前体验结论

### 优势

- 页面非常安静，没有广告、侧栏和复杂推荐流，阅读目标清楚。
- `随便聊聊` 的语气比“技术文章”更自然，符合个人博客气质。
- 文章列表采用文本流而不是卡片流，适合长期归档和连续浏览。
- 日期、标题、摘要的层级清楚，去掉重复日期图标后阅读更顺。
- 顶部导航简单，文章、摄影、标签、关于四个入口心智稳定。
- 低饱和米白背景、深灰正文、蓝灰辅助文字、暖色强调线整体协调。

### 不足

- 首屏语境偏弱，`随便聊聊` 下方缺少一句帮助新访客理解内容范围的说明。
- 桌面端从导航到标题、再到第一篇文章之间留白较大，气质好但信息效率偏低。
- 移动端标题视觉重量较强，第一篇文章出现位置偏下。
- 内容发现路径单一，目前主要依赖时间线，缺少高频标签或主题入口。
- 普通内容页 hero eyebrow 前的小图标不是必要信息，可以继续减少装饰感。
- 文章数量继续增长后，`/articles/` 全量展示会逐渐变长，需要提前记录分页触发条件。

## 3. 改进方案

### P1：首屏语境与留白收口

建议：

- 在 `随便聊聊` 下增加一句克制说明：`工程实践、工具折腾和一些不太正式的记录。`
- 略微压缩 `/articles/` hero 区域上下留白，优先照顾移动端。
- 移除 `/articles/` 普通内容页 hero eyebrow 前的小图标，只保留 `ARTICLES` 文字标签。
- 摘要文字略微加深，提高手机屏幕上的可读性。

可判定验收：

- `1440x900`：首屏可见页面标题、说明文案和第一篇文章的日期或标题开头。
- `390x844`：首屏可见 `随便聊聊`、说明文案和第一篇文章标题开头。
- `375x667`：首屏至少可见页面标题、说明文案和第一篇文章日期。
- 任意测试视口中 `document.documentElement.scrollWidth === document.documentElement.clientWidth`。
- 长中文标题、英文技术词和下划线标识符自然换行，不制造横向滚动。

### P1：移动端阅读密度优化

建议：

- 移动端 `随便聊聊` 标题字号可小幅降低，但保留当前文学感。
- 文章 item 的上下 padding 可以小幅收紧，但不能变成信息流或新闻列表。
- 如果修改共享 `.entry-*` 样式，必须同步回归首页近期文章、标签结果页和分页页。

可判定验收：

- `375x667`、`390x844`、`768x1024` 下标题和摘要均无截断、无横向滚动。
- 浅色和深色主题下，日期、标题、摘要、分割线对比度稳定。
- 首页近期文章区、任一标签页、`/page/[page]` 没有因为共享样式改动出现断层。

### P2：内容发现增强

建议：

- 在文章列表上方或标题说明下方增加 5-8 个高频标签入口。
- 标签入口应轻量、低饱和、文本优先，不做动态标签云。
- 标签入口换行展示，不做横向滚动，也不使用持续动画。

数据规则：

- 数据源：`getCollection('posts', ({ data }) => !data.draft && data.type !== 'photography')`。
- 统计字段：`post.data.tags`，忽略空值或非字符串 tag。
- 排序：先按 count 降序，同 count 时使用 `tag.localeCompare('zh-CN')` 固定顺序。
- 数量：最多展示 8 个；如果有效标签少于 3 个，则隐藏这一行或只提供“全部标签”入口。
- 链接：使用 `/tags/${encodeURIComponent(tag)}/`，特殊字符和中文标签不能破坏 URL。
- 可访问性：标签是普通链接，移动端触控目标建议接近 44px，focus ring 必须可见。

可判定验收：

- 新访客可以通过标签快速进入主题，而不只依赖时间线。
- 标签入口不会抢文章列表主视觉。
- 标签数量增加后仍可换行显示，不引入横向滚动。
- draft 和摄影内容不会被统计进文章页高频标签。

### P2：文章列表增长策略

建议：

- 当前文章数量继续全量展示可接受。
- 当非摄影文章超过 30 篇后，重新评估 `/articles/` 是否改为分页第一页或加入明确分页入口。
- 不做 infinite scroll，保持静态站简单、SEO 友好、可维护。
- 不直接复用当前 `/page/[page]` 心智作为文章归档分页，除非同时明确首页分页和文章分页的关系。

可判定验收：

- 发布前可以统计当前非摄影文章数量。
- 如果启用分页，需验证 canonical、sitemap、页码链接和重复索引风险。
- `/articles/` 不成为过长页面，分页入口保持静态链接和清晰页码。

## 4. Sr. Front-end 实施说明

### 目标文件

- `src/pages/articles.astro`：添加 hero 说明文案、计算高频标签入口、必要时移除该页 hero eyebrow 图标。
- `src/styles/global.css`：仅在确认为共享排版收益时修改 `.entry-*`、`.section-*` 等全局样式；否则优先使用文章页局部 class 降低回归风险。
- `src/layouts/PostLayout.astro`：本轮不主动修改，除非后续明确要调整文章详情页 meta 图标。若调整，只移除装饰性 calendar 图标，不改变日期语义。

### 实施边界

- 不引入 CMS。
- 不引入 infinite scroll。
- 不增加新运行时依赖。
- 不修改摄影页固定深色主题策略。
- 不把文章列表改成卡片流。

### 跨页面回归范围

如果修改全局文章列表样式，必须检查：

- `/articles/`
- `/`
- 任一 `/tags/[tag]/`
- 现有分页页，例如 `/page/2/` 如果存在
- 任一普通文章详情页
- `/photography/` smoke test，确认摄影深色系统没有被普通页样式污染

## 5. QA 与 Code Review 验收矩阵

### 命令门禁

- `npm run build`
- `npm run check:content-health`
- `npm run check:security`
- `git diff --check`

### 视口矩阵

- 桌面：`1440x900`
- iPad：`768x1024`
- iPhone：`390x844`
- 小屏手机：`375x667`

### 主题矩阵

- 普通内容页浅色主题。
- 普通内容页深色主题。
- 摄影页固定深色主题 smoke test。
- 主题切换后刷新和跨页面跳转状态保持稳定。

### 内容样本

- 长中文标题。
- 长英文技术词。
- 下划线标识符，例如 `use_case`。
- 无摘要文章。
- 超长摘要文章。
- 中文标签、英文标签、带空格标签、带特殊字符标签。

### 无障碍验收

- 键盘 Tab 顺序自然。
- 标签入口和导航链接 focus 可见。
- 装饰性图标使用 `aria-hidden` 或不进入可访问名称。
- 日期不被重复朗读。
- 文本颜色在浅色和深色主题下有足够对比度。

### Code Review 检查点

- 改动范围与文章页专项一致。
- 高频标签统计排除 draft 和摄影内容。
- 标签排序稳定，链接 URL 编码正确。
- 全局样式改动有对应跨页面回归记录。
- 没有引入重型依赖、动态标签云或无限滚动。

## 6. 上线回归顺序

1. 本地完成实现后运行命令门禁。
2. 本地 preview 或构建产物中检查 `/articles/` 桌面、iPad、手机视口。
3. QA 检查普通文章页、标签页、首页和摄影页 smoke test。
4. Code Review 确认 scope、可维护性和无明显回归风险。
5. 提交并 push。
6. 部署完成后访问 `https://notes.miaowu.org/articles/` 做线上回归。
7. 线上回归至少确认：首屏层级、移动端无横向滚动、主题切换状态、标签入口可点击、普通文章详情页可访问。
