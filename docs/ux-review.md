# 博客真实体验审计

**更新日期**：2026-04-19  
**审计目标**：从 Product、UI/UX、普通用户 / Developer 三个视角，评估当前博客的阅读体验、摄影浏览体验和内容添加体验。

## 线上观察补充

**观察时间**：2026-04-19  
**观察页面**：

- `https://notes.miaowu.org/`
- `https://notes.miaowu.org/articles/`
- `https://notes.miaowu.org/photography/`
- `https://notes.miaowu.org/posts/2024-10-01-photography-b6hlboogdju/`

实际打开线上页面后，有几处需要修正或提高优先级：

- 普通页面的深色主题会被保留，说明主题持久化已经在真实环境生效；这让文章页和摄影页在视觉上更连续，但也要求普通深色主题的文章可读性持续稳定。
- 摄影列表页归档卡片在可访问树里出现重复日期，说明信息重复不只是视觉层面的问题，也会影响辅助技术读取体验。
- 无标题摄影作品详情页的浏览器标题显示为 `undefined | Joey's Notes`，这是专业感、分享标题和 SEO 都会受影响的 P0 问题。
- 摄影详情页底部作品胶片条方向正确，能够替代传统“较新 / 较早”按钮；但当前作品和可滑动区域的提示还可以再明确。
- 文章列表页 `/articles/` 确认为全量文章列表，当前数量可接受；内容量增加后需要引入分页入口或改为分页第一页。

## 结论摘要

当前博客已经从“技术文章站”演进为“文章 + 摄影作品集”的双主线静态站。文章阅读体验相对稳定，摄影展示体验有明显记忆点，Markdown + Astro 的内容结构也利于长期维护。

主要问题不在功能缺失，而在两个方面：

- 内容消费：摄影列表与详情页需要继续降低文字和控件干扰，让图片成为第一主角。
- 内容生产：文章和摄影都依赖手写 frontmatter，摄影字段更多，发布体验仍偏工程化。

## Product 视角

### 优势

- 文章和摄影已经有清晰区分：文章服务“阅读与查阅”，摄影服务“观看与连续浏览”。
- 信息架构简单：首页、文章、摄影、标签、关于构成了稳定的长期归档结构。
- Astro 静态站降低了运维复杂度，适合个人博客长期维护。
- 摄影详情页已经具备作品集特征：深色沉浸、灯箱、缩略图导航、作品胶片条。

### 不足

- 首页需要持续避免“文章区 + 摄影区简单拼接”，否则双主线会显得割裂。
- 摄影列表页仍承担较多信息展示任务，容易从“作品归档”滑向“内容卡片列表”。
- 标签系统可用，但标签对新访客的引导价值有限，需要依赖内容量增长后再判断是否强化。
- 内容增长后，摄影单页归档可能变长，未来需要考虑年份独立页或更明确的分段浏览。

### 建议

- 保持内容页和摄影页两套清晰心智：文章偏阅读，摄影偏观看。
- 首页只保留必要入口，不为设计解释设计。
- 摄影列表页继续减少说明性文案和重复元信息。
- 将“发布体验”作为产品能力看待，而不是只作为开发流程。

## UI/UX 视角

### 阅读体验

文章页目前的行宽、留白和流式列表是合理的。`reading-narrow`、`entry-stream`、`entry-item` 这类排版原语让普通文章阅读相对稳定。

优势：

- 正文行宽适合长文阅读。
- 文章列表弱卡片化，接近文本流，符合当前博客语气。
- 普通页面支持浅 / 深主题切换，摄影页锁定深色，规则清晰。

不足：

- `/articles/` 当前展示全部非摄影文章，随着文章增加会变长。
- `/page/[page]` 有静态分页，但 `/articles/` 本身没有分页入口策略。
- 部分文章历史内容的标题、代码块、图片质量依赖原始 Markdown，需要持续内容治理。

### 摄影浏览体验

摄影详情页是当前最有特色的部分。全屏首图、黑底、灯箱、作品内缩略图和底部作品胶片条方向正确。

优势：

- 深色主题固定，视觉稳定。
- 首图全屏展示，进入作品后有沉浸感。
- 同一篇多图作品支持缩略图、灯箱、滑动切换和当前图定位。
- 底部作品胶片条比传统“较新 / 较早”翻页更适合连续浏览摄影作品。

不足：

- 摄影列表页仍需警惕信息密度，日期、相机、标题、按钮都可能抢图。
- 移动端需要作为第一验收设备，尤其是 iPhone 上的菜单、灯箱、横向滑动和底部胶片条。
- 横向滑动控件需要足够明显，但不能增加强遮罩或高成本动效。

### 建议

- 摄影列表页优先让图片表达，不要用过多文字解释作品。
- 摄影详情页的浮层继续遵循“可读但不打扰照片”的原则。
- 横向滑动区域可以增加轻量提示，但避免持续动画。
- 每次摄影 UI 调整都要单独检查手机、iPad、桌面三种宽度。

## 普通用户 / Developer 视角

### 内容添加体验

当前内容存放在 `src/content/posts/*.md`，schema 定义在 `src/content/config.ts`。文章和摄影共用 collection，通过 `type` 区分。

优势：

- Markdown 文件可读、可审查、可版本控制。
- Git 提供天然回滚能力。
- `draft` 字段能控制是否发布。
- 已有 `photos:sync`、`check:content-health`、`fix:content-health` 等脚本，说明内容流程已有自动化基础。

不足：

- 发布文章需要记住 frontmatter 字段。
- 发布摄影需要处理图片、路径、EXIF、gallery、camera、publishedDate，心智负担更高。
- 手机端发布不顺手；当前更适合在电脑上手动编辑和提交。
- `photos:sync` 可以生成摄影文章，但还需要更清晰的使用 SOP 和结果校验步骤。

### 建议

- 短期继续手动发布，不引入 CMS。
- 为文章和摄影分别建立稳定发布 SOP。
- 后续可以增加 `new:post` / `new:photo` 脚本生成模板，减少手写 frontmatter。
- 内容健康检查应作为发布前必跑检查。

## 页面级审计

| 页面 | 当前优势 | 主要风险 | 优先级 |
|---|---|---|---|
| 首页 | 双主线入口清晰 | 文章与摄影可能割裂 | P1 |
| 文章列表 | 文本流清爽 | 全量展示未来会变长 | P1 |
| 文章详情 | 阅读宽度稳定 | 历史 Markdown 质量不一 | P2 |
| 标签总览 | 静态权重标签性能稳定 | 对新访客引导弱 | P2 |
| 标签详情 | 过滤路径明确 | 与文章列表体验接近，可继续统一 | P2 |
| 摄影列表 | 深色摄影系统已建立 | 信息密度和卡片感要继续控制 | P0 |
| 摄影详情 | 沉浸感强，交互丰富 | 无标题作品标题 fallback、移动端控件密度需持续验证 | P0 |

## 验收清单

- 普通内容页主题切换跨页保持，摄影页固定深色。
- iPhone 上摄影详情页能打开导航，能退出作品，能操作灯箱。
- 摄影列表页首屏不出现明显色块断层或多余解释文案。
- 摄影列表页归档卡片不重复朗读日期。
- 无标题摄影作品不能在浏览器标题里显示 `undefined`。
- 文章页阅读不需要横向滚动，代码块不会出现异常底色或选中感。
- 添加文章和摄影时，有明确 SOP 可跟随，不依赖记忆。
